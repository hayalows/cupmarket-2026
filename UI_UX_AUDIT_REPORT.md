# CupMarket UI/UX Audit

Date: 2026-06-13
Branch: `codex/cupmarket-audit`
Repository: `hayalows/cupmarket-2026`

## Executive Summary

CupMarket currently works as a clear analytics dashboard for World Cup data,
model probabilities, group tables, and virtual country prices. It is useful,
stable, and understandable for someone who already knows the model.

It does not yet feel like a polished fantasy trading product. The interface has
no account state, no portfolio, no buy/sell journey, no order confirmation, no
market status, no visible fantasy-credit balance, and no product-level visual
system. The current experience says "model dashboard"; the target experience
should say "premium fantasy exchange for World Cup countries."

The next design direction should preserve the strong analytics foundation but
reframe it around user decisions:

- What should I buy?
- Why is this country priced this way?
- What do I own?
- How much am I up or down?
- Is the market open?
- What changed since the last price update?

## Audit Method

The audit reviewed:

- `app.py` UI structure and navigation.
- Rendered Streamlit app at desktop width.
- Rendered Streamlit app at mobile viewport size.
- Streamlit testing output for every page.
- Current data artifacts and table density.
- Product readiness for fantasy trading accounts and market flows.

## Current Experience Strengths

### Strong data foundation

The app already has the hard part: meaningful live-ish World Cup data, model
probabilities, generated prices, group tables, and model-health metadata. That
is a strong product core.

### Clear page separation

The current sections are understandable:

- Overview
- Match Centre
- Country Market
- Team Explorer
- Group Tables
- Model Health
- How It Works

For an analytics dashboard, that structure is sensible.

### Good operational transparency

The app shows whether scores come from the saved snapshot or live API, when the
simulation was generated, and that football-data.org is the data provider. That
is good trust-building material.

### Useful model-health page

The model-health page gives evidence that the market prices are generated by a
real model, not arbitrary numbers. This is valuable, especially for advanced
users.

## Core Product Gap

CupMarket is intended to be a fantasy trading app, but the current UI does not
yet support the fantasy trader mental model.

Today, the user can observe.

The target product needs the user to act:

- sign up
- receive fantasy credits
- browse tradable countries
- buy shares
- sell shares
- track holdings
- compare performance
- learn why prices moved
- compete on a leaderboard

The current dashboard should become the analytics layer inside a larger trading
product, not the whole product surface.

## Information Architecture Findings

### Finding 1: Navigation is analytics-first, not trading-first

The first navigation item is "Overview," and the core product action is absent.
For a fantasy trading product, the main navigation should be centered on market
activity and ownership.

Recommended future navigation:

1. Market
2. Portfolio
3. Matches
4. Leaderboard
5. Watchlist
6. Learn
7. Admin or Model Health

`Model Health` should probably move out of the primary user navigation. It can
remain available as an advanced transparency page, but it should not compete
with trading flows.

### Finding 2: "Country Market" is the real product center

The `Country Market` page is currently one of several tabs, but it should become
the home of the product. It should contain:

- market status
- user fantasy balance
- sortable country list
- buy and sell actions
- price movement
- user's owned quantity
- watchlist controls
- short explanation of price

### Finding 3: The app needs object-level pages

`Team Explorer` is the start of a country-detail page. In the future, each
country should have a dedicated route/page:

- current price
- buy/sell ticket
- user holdings
- probability ladder
- fixtures and recent results
- price history
- model explanation
- risk notes

This is where a premium product can feel calm, confident, and useful.

## Visual Design Findings

### Finding 4: Current visual system is default Streamlit

The current interface uses Streamlit defaults. It is clean enough for a
prototype, but it does not have a distinctive CupMarket product identity.

Target visual direction:

- calm premium sports-finance aesthetic
- high-contrast readable tables
- restrained color palette
- strong use of whitespace
- country identity accents through flags/colors, not noisy backgrounds
- clear green/red movement language for gains/losses
- refined card surfaces for repeated market assets

Avoid:

- decorative gradients as the primary identity
- oversized marketing-style hero sections inside the app
- dense raw data tables as the first thing users see
- too many equally weighted metrics

### Finding 5: Hierarchy needs more product storytelling

The current Overview shows:

- finished matches
- live matches
- upcoming matches
- market leader

For a fantasy trading app, the first viewport should instead answer:

- What is my portfolio worth?
- What changed today?
- What are the top opportunities?
- Is trading open?
- What is the next match that could move prices?

Recommended first viewport after login:

- Portfolio value
- Available credits
- Today's P/L
- Market status
- Top movers
- Watchlist
- Next price-moving matches

### Finding 6: Tables are useful but too raw for ordinary users

The market table exposes many columns at once. Advanced users may like this,
but new users need a more scannable trading surface.

Recommended market row/card fields:

- country
- group
- current price
- price change
- win probability
- reach Round of 32 probability
- user's holding quantity
- buy/sell buttons

Advanced probability columns should be available through expand/detail views,
not always shown by default.

## Interaction Design Findings

### Finding 7: No primary action

The current app has almost no action besides filtering, refreshing, selecting a
team, and reading tables. A trading app needs an obvious action system.

Primary actions:

- Buy
- Sell
- Add to watchlist
- View country

Secondary actions:

- Sort
- Filter
- Compare
- View model explanation

### Finding 8: No trade ticket

The product needs a focused trade ticket pattern:

- selected country
- current buy price
- current sell price
- quantity stepper/input
- estimated cost/proceeds
- available credits
- resulting position
- confirmation button
- fantasy-only disclaimer

The trade ticket should be server-validated. The UI can guide, but it must not
be the source of truth for balance or holdings.

### Finding 9: No feedback loop after user action

Future buy/sell actions need immediate feedback:

- success state
- updated balance
- updated holding
- transaction record
- error state if market is halted or funds are insufficient

This feedback should be precise, not generic.

## Fantasy Trading UX Rules

The interface should reinforce these rules:

1. CupMarket is fantasy only; credits have no cash value.
2. Each country is a fantasy share/contract.
3. The model price is the expected tournament settlement value.
4. Users start with a fixed fantasy balance.
5. Users cannot have negative cash.
6. Users cannot sell more shares than they own.
7. No short selling in V1.
8. No leverage or margin in V1.
9. Market buy/sell is enough for V1.
10. Limit orders can wait for V2.
11. CupMarket acts as the market maker in V1.
12. Trading should halt during price updates.
13. Country trading may halt during that country's live match if API delay
    creates unfair information risk.
14. Every transaction must be permanently recorded.
15. Final tournament settlement converts holdings back into fantasy credits.

## Page-by-Page UX Audit

### Overview

Current role:

- Summary dashboard.

Problems:

- Does not show user portfolio state.
- Does not explain what the user should do next.
- Market leader is useful but not actionable.
- "What the leader's probability means" is educational but not a strong product
  moment.

Recommendation:

Turn Overview into a signed-in Home page:

- portfolio value
- fantasy credit balance
- daily P/L
- market status
- top movers
- watchlist
- next matches
- recommended "Explore Market" action

### Match Centre

Current role:

- Fixture/results table with model outputs.

Problems:

- It is data-rich but hard to scan.
- Match importance is not ranked.
- It does not connect matches to market impact.

Recommendation:

Reframe as "Matches that move the market":

- live/next/upcoming sections
- affected countries
- current country prices
- estimated volatility/importance
- link to trade country

### Country Market

Current role:

- Full price/probability table and top-20 bar chart.

Problems:

- Best product page is still mostly a data table.
- No buy/sell affordance.
- No holding/watchlist state.
- Percentages are partly raw decimals in this page.

Recommendation:

Make this the core trading screen:

- compact market list
- segmented filters: All, Watchlist, Movers, Undervalued, Favorites
- search country
- sort by price, change, champion probability, group
- buy/sell buttons
- country detail drawer/page

### Team Explorer

Current role:

- Single-country analytics.

Problems:

- Useful but visually sparse.
- No price history context above the fold.
- No trading action.
- Team selection is a generic dropdown rather than a product object page.

Recommendation:

Turn into country detail:

- country header with flag and group
- current price and change
- buy/sell ticket
- user's holdings
- probability ladder
- upcoming matches
- price history
- model explanation in plain language

### Group Tables

Current role:

- Twelve group tables.

Problems:

- Long vertical list.
- Valuable for tournament context but not a primary trading workflow.

Recommendation:

Use tabs/segmented group selection, or show group table inside country detail.
The full page can remain for advanced users, but it should be visually lighter.

### Model Health

Current role:

- Model transparency page.

Problems:

- Too technical for ordinary fantasy users.
- Good for trust, but too prominent in primary nav.

Recommendation:

Move to "About the model" or "Transparency." Keep it accessible but separate
from the trading path.

### How It Works

Current role:

- Explains the system.

Problems:

- Explains the model pipeline more than the fantasy trading game.
- The no-real-money warning is correct but should be part of a clearer product
  rules section.

Recommendation:

Split into:

- How CupMarket trading works
- How prices are calculated
- Fantasy-only rules
- Tournament settlement values
- Data/model limitations

## Mobile UX Audit

The mobile viewport rendered without exceptions, but the experience is not yet
mobile-first.

Key issue:

- Sidebar navigation dominates the mobile experience.
- The app content appears after navigation/status metadata.
- Data tables are not ideal as the primary mobile trading interface.

Mobile recommendation:

- Use bottom navigation or top tabs in a future Next.js app.
- Show a compact market list, not full-width analytical tables.
- Use country cards/rows with price, change, and one clear action.
- Move model/data metadata below primary content.
- Use persistent portfolio balance and market status at the top.

## Accessibility Findings

Strengths:

- Streamlit provides basic semantic structure.
- Navigation is keyboard-accessible through standard controls.
- Tables have built-in controls.

Risks:

- Price movement should not rely on color alone.
- Dense tables can be hard for screen readers.
- Long page structures need clearer landmarks in a custom app.
- Buy/sell actions need explicit labels and confirmation states.

Recommendations:

- Always pair color with sign/text, for example `+2.4%` and `Up`.
- Use clear button labels: `Buy Spain`, `Sell Spain`.
- Keep touch targets at comfortable mobile sizes.
- Use visible focus states.
- Provide plain-language explanations for model outputs.

## Product Design North Star

CupMarket should feel like:

> A premium fantasy exchange where football fans trade World Cup countries with
> virtual credits and understand why prices move.

Design values:

- Calm, not chaotic.
- Trustworthy, not casino-like.
- Fast to scan, deep when needed.
- Beautiful, but not decorative at the expense of clarity.
- Game-like, but clearly fantasy-only.

The future Next.js app should be organized around product workflows:

```text
/
  Home dashboard
/market

/market/[country]

/portfolio

/matches

  Optional model/data operations view
```

## Design System Direction

### Typography

Use a system type stack that feels native and crisp:

```css
font-family:
  Inter,
  ui-sans-serif,
  system-ui,
  -apple-system,
  BlinkMacSystemFont,
  "Segoe UI",
  sans-serif;
```

For an Apple-like feel:

- use strong typographic hierarchy
- avoid oversized type inside dense app panels
- use tabular numbers for prices and portfolio values
- reserve large type for true hero/portfolio numbers

### Color

Recommended direction:

- neutral off-white/dark text base
- deep charcoal for primary text
- restrained green for gains
- restrained red for losses
- gold or electric accent for CupMarket identity
- country color/flag accents only in small doses

Avoid making the entire app green/red or dark-blue finance themed. The product
should feel sports-finance premium, not generic trading-terminal.

### Components

Needed product components:

- Market row
- Country card
- Price chip
- Change chip
- Trade ticket
- Portfolio summary
- Holding row
- Transaction row
- Market status banner
- Country probability ladder
- Settlement value table
- Match impact card
- Empty state
- Error state
- Loading state

## Prioritized Redesign Roadmap

### Phase 1: Product Definition

- Finalize fantasy trading rules.
- Define settlement values.
- Define market hours/halt rules.
- Define portfolio accounting.
- Define all user states: logged out, new user, active trader, settled season.

### Phase 2: Data and Auth

- Add Supabase schema for users, profiles, portfolios, holdings, transactions,
  price snapshots, and public market data.
- Keep Python as the model engine.
- Write model outputs into Supabase.
- Use Supabase Auth for signup/login.

### Phase 3: Next.js Trading MVP

- Build market list.
- Build country detail page.
- Build buy/sell trade ticket.
- Build portfolio page.
- Build transaction history.
- Build fantasy-only rules page.

### Phase 4: Premium UX

- Add price charts.
- Add top movers.
- Add watchlists.
- Add leaderboard.
- Add match-impact cards.
- Add polished responsive design.

### Phase 5: Advanced Market Features

- Add limit orders.
- Add private leagues.
- Add alerts.
- Add season archive.

## Recommended Immediate UI Changes if Staying in Streamlit Temporarily

If Streamlit remains the short-term demo surface, make these improvements before
moving fully to Next.js:

1. Rename `Country Market` to `Market`.
2. Move `Market` to the first navigation item.
3. Add a top fantasy disclaimer: "Fantasy credits only. No real money."
4. Format all market percentages consistently.
5. Add a simplified market table view with fewer default columns.
6. Add a placeholder portfolio summary to frame the future product.
7. Add a `Trading Rules` page.
8. Move `Model Health` below `How It Works` or into an advanced section.

## Final Recommendation

Do not try to make Streamlit become the final consumer trading product.

Use Streamlit as:

- internal analytics
- model monitoring
- admin dashboard
- quick prototype surface

Use Next.js as:

- polished public app
- authentication layer
- fantasy trading interface
- portfolio experience
- mobile-first product surface

The current Python/model system should remain the brain of CupMarket. Supabase
should become the memory. Next.js should become the polished product experience.
